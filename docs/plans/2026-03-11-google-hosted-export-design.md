# Google Hosted Export Design

## Goal
Add a first-class `google-hosted` export target so the same project workspace can be published for Google Classroom delivery with Google sign-in and cross-device save/resume, without changing the source workspace files.

## Product Direction
The repo should support one source project with multiple delivery targets:
- `scorm` for Brightspace resume and LMS tracking
- `google-hosted` for Google Classroom delivery with Google account resume
- `single-html` for static local/offline handoff when no identity-backed persistence is required

The Google-hosted target should feel like the existing export flows:
- one command
- one export folder
- predictable output location
- no manual editing inside the generated bundle

## User Model
- Teacher exports a project as `google-hosted`.
- Teacher publishes the exported bundle to a Google-managed Firebase project.
- Teacher posts the hosted URL in Google Classroom.
- Student opens the activity from Classroom.
- Student signs in with their school Google account the first time.
- Student answers questions, leaves, returns on another browser or computer, and resumes automatically.
- Student prints the existing report from inside the app when finished.

## Why Google Sites Alone Is Not Enough
Google Sites can link to or embed web content, but it does not provide an LMS-style runtime or a cross-device learner state API.

That means:
- a raw static export embedded in Sites can only rely on browser storage
- browser storage does not follow the student across devices
- the persistence layer must come from an identity-backed backend, not from Sites itself

For this repo, the right host/runtime pair is:
- Firebase Hosting for the app
- Firebase Authentication with Google sign-in for identity
- Firestore for per-student saved state

## V1 Decision
Version 1 should ship a `google-hosted` export bundle only.

That means:
- the repo generates a ready-to-deploy Firebase app bundle
- the repo does not automate Firebase deployment yet
- deployment remains a documented manual step using the Firebase CLI

This keeps the first implementation aligned with the existing export architecture and avoids coupling the repo to local developer auth, Firebase project secrets, or account-specific deployment state.

## Export Surface
CLI:
- `npm.cmd run export:google-hosted -- --project <slug>`

Studio:
- add a `Google Hosted` export action beside the existing export commands

Output:
- `projects/<slug>/exports/google-hosted/`

The export should be deterministic:
- clear the previous `google-hosted` export folder
- copy the workspace into the new export folder
- inject the Google-hosted runtime bridge
- generate Firebase config placeholders and deployment docs

## Bundle Shape
The `google-hosted` export folder should contain:
- the copied workspace entrypoint and assets
- `google-hosted-bridge.js`
- `firebase-config.template.json`
- `firebase.json`
- `.firebaserc.template`
- `README-deploy.md`

Optional support files are acceptable if they are generated by the export command and live only inside `projects/<slug>/exports/google-hosted/`.

## Runtime Architecture
The source workspace remains untouched.

At export time, the generated Google-hosted runtime should be injected before the project's local script entrypoint, similar to the SCORM bridge pattern.

The runtime bridge is responsible for:
- initializing Firebase using generated config
- signing the student in with Google
- restoring saved state after auth completes
- saving state on changes and navigation
- exposing a narrow persistence contract to the exported app

The workspace code should not be rewritten around Firebase internals. The export bridge should adapt the existing local state shape to a cloud-backed store.

## Persistence Model
Saved state should be keyed by:
- project slug
- Google user ID

Recommended Firestore path:
- `projects/{slug}/users/{uid}`

Stored document shape:
- `projectSlug`
- `userId`
- `savedAt`
- `state`
- `reportSnapshot` when the project already maintains one in local state
- `schemaVersion`

The runtime should preserve the existing per-project local state structure inside `state`, not invent a new answer model.

## Auth Flow
Use popup-based Google sign-in in version 1.

Why:
- the app is launched from Google Classroom
- students are already likely signed into Google
- popup auth is more reliable than redirect auth in embedded/cross-origin contexts
- the export target is meant to be linked from Classroom, not embedded inside Google Sites

The student experience should be:
- first launch: `Sign in with Google`
- later launches: automatic session restore when possible

## UI Expectations
The Google-hosted runtime may add a small fixed control surface, but it should stay lighter than the SCORM overlay.

Required UI behaviors:
- signed-out state with a clear `Sign in with Google` action
- signed-in state with autosave status
- restore state silently after sign-in
- do not interfere with the project's printable report UI

## Security And Privacy
Version 1 should avoid collecting anything beyond what the project already stores for its own student state.

Recommended rules:
- only authenticated users can read/write their own project state
- no teacher dashboard in version 1
- no student-to-student visibility
- no custom role model yet

If domain restriction is needed later, it should be configuration-driven and not hard-coded into the export logic.

## Studio And Server Integration
The new export target should behave like existing exports:
- command type added to shared export command types
- local server command runner maps it to the CLI script
- Studio export actions can trigger it without special-case UI architecture

## Documentation Requirements
If implemented, update:
- `README.md` for the new export command and teacher workflow
- `ARCHITECTURE.md` for the new Google-hosted export boundary and runtime responsibilities

## Out Of Scope
- automated Firebase deployment from the repo
- Google Classroom grade passback
- teacher dashboards or student monitoring
- Google Sites embed-specific support
- offline-first resume
- replacing SCORM or Brightspace export paths

## Risks
- Firebase setup is a one-time prerequisite outside the repo
- popup auth behavior can vary with school browser policy
- projects with very custom local state may need targeted storage-key detection adjustments
- export documentation must be clear enough for non-developer publishing steps

## Success Criteria
- `npm.cmd run export:google-hosted -- --project <slug>` produces a complete bundle in `projects/<slug>/exports/google-hosted/`
- the exported bundle can be deployed to Firebase Hosting without editing workspace source files
- a student can sign in with a Google school account, answer content, leave, return on another browser or computer, and restore their state
- the printable report still works in the hosted build
- the same source project can still export to SCORM without regression
